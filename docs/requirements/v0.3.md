# EB塾 指導管理 Web アプリ｜要件定義 v0.3

> 現状の Google Sheets「★新【EB塾】【2026年度】指導モニタリング管理表」を Web アプリへ移行するための要件定義。
> v0.1 → v0.3 で PO（原口）からの確認回答を反映済み。Phase -1（Sheets パターン抽出）は完了し、Phase 0（設計確定）への入力。

---

## 0. 技術スタック

| 層 | 採用 |
|---|---|
| 言語 | Python 3.12+ |
| Web | FastAPI |
| UI | Jinja2 + HTMX + Tailwind |
| DB | Supabase (Postgres) — RLS でロール別アクセス制御 |
| 認証 | Supabase Auth（招待メール方式） |
| ストレージ | Google Drive（教師契約書 PDF）+ Supabase Storage（その他） |
| デプロイ | Cloud Run（Gunicorn + UvicornWorker） |
| CI/CD | GitHub Actions → Artifact Registry → Cloud Run |
| 監視 | Cloud Logging / Error Reporting |

判断軸: **業務系 LOB ／ 同時編集少 ／ 帳票・集計多 ／ 関係者≤数十名** → SPA 不要、HTMX で十分。

## 1. スコープ

### IN
- 教師・生徒・指導枠（選考番号）・月次レポート・面談議事録の CRUD
- 契約時間 vs 実施時間の月次集計と充足率算出
- 月次報告書／面談記録の未提出アラート
- 教師／管理者／経理 のロール別ビュー
- スプレッドシートとの並行運用期間（3か月）

### OUT（フェーズ2以降）
- 保護者ポータル（生徒・保護者ログイン）
- 請求・入金管理
- レッスン予約／カレンダー連携
- AI による議事録要約・進捗予測

## 2. ロール

| ロール | 割当方法 | 主な権限 |
|---|---|---|
| 管理者 | `users.role='admin'` 付与 | 全データ R/W、設定変更、教師・生徒登録 |
| 教師 | **招待メール方式**（管理者が招待 → 教師がリンクで初回サインイン） | 自担当生徒のみ R/W |
| 経理 | `role='accounting'` | 教師契約・口座情報・契約時間集計の閲覧 |
| 生徒/保護者 | フェーズ2 | — |

## 3. 機能要件（FR）

### FR-1 教師管理
- F-1.1〜3 教師の登録／編集／無効化（論理削除）
- F-1.2 列: 教師番号 / 氏名 / 大学・学部 / IBスコア / HL・SL科目 / 指導言語 / 契約書状況 / 口座情報 / 業務報告アカウント / 研修ステータス / 稼働状況 / おすすめポイント
- F-1.3 担当生徒一覧の逆引き表示
- F-1.6 **招待メール方式**:
  - 管理者が `teachers` レコード作成（email 含む）
  - 「招待を送信」で Supabase Auth `inviteUserByEmail()` 実行
  - 招待トークン有効期限: 72時間（再送可）
  - ステータス: `invited` → `active` → `disabled`
- F-1.7 招待メールから初回サインインで `users.teacher_id` を自動紐付け
- F-1.8 セルフサインアップは不可（招待トークン必須）

### FR-2 生徒管理
- F-2.1 生徒の登録／編集／卒業ステータス変更
- F-2.2 列: 生徒番号 / 氏名 / 学年 / 最終試験時期 / 入会日 / 志望大学 / 志望分野
- F-2.3 科目別の担当教師サマリ

### FR-3 指導枠（選考番号）
- F-3.1 1生徒 × 1科目 × 1教師 で1レコード（選考番号 = 一意ID）
- F-3.2 列: 指導コース / 科目 / 月次契約時間 / 開始月 / 終了月 / ステータス
- F-3.3 教師変更履歴を保持（タイムライン）

### FR-4 月次レポート
- F-4.1 選考番号 × 年月 で1レコード
- F-4.2 入力項目: 実施時間 / 充足率（自動）/ 指導報告書チェック / 生徒達成度 / 教師達成度 / 計画性 / キャンセル / 総合満足度 / ユニットテストスコア
- F-4.3 月初に未入力レコード自動生成、月末締切前に通知

### FR-5 面談記録
- F-5.1 種別: 三者面談 / 教師MTG / その他
- F-5.2 列: 日付 / 種別 / 参加者 / トピックタグ（複数）/ 本文 / ToDo
- F-5.3 ToDo は子テーブル（期限・担当・状態）
- F-5.4 全文検索＋タグフィルタ
- F-5.5 **教師は自担当生徒（assignments）の議事録のみ閲覧**。他教師が記入した議事録でも、当該生徒担当でなければ不可視
- F-5.6 管理者・経理は全件可視

### FR-6 ダッシュボード
- F-6.1 管理者: 未提出レポート／未実施面談／充足率異常値
- F-6.2 教師: 自分の今月の担当・締切・ToDo
- F-6.3 「自分宛て」を最上段に

### FR-7 通知
- F-7.1 月次報告書未提出 → 締切前に段階通知（事前リマインド日数は設定で可変）
- F-7.2 ToDo 期限超過 → 担当者に通知

### FR-8 取込・出力（並行運用）
- F-8.1 既存スプレッドシートからの初期インポート
- F-8.2 月次集計の CSV / Excel 出力
- F-8.3 並行運用期間（3か月）中、各エンティティに `external_sheet_url` を持たせて元 Sheets 該当行へ 1 クリック遷移
- F-8.4 並行運用終了 → Sheets 凍結（読み取り専用）→ 1か月後に廃止
- F-8.5 自動パーサーは Phase -1 仕様書（[docs/sheets-migration/parser-spec.md](../sheets-migration/parser-spec.md)）に従い実装
- F-8.6 自動パーサーは Sheets API → Pydantic 検証 → DB UPSERT
- F-8.7 解釈不能行は `import_errors` テーブルに溜め、管理者画面で手動修正

### FR-9 設定管理
- F-9.1 管理者のみアクセス可能な「設定」画面
- F-9.2 設定項目（`system_settings` テーブル）:
  - 月次レポート締切日（当月末／翌月N日 等、可変）
  - 充足率異常値の上限・下限（デフォルト 80% / 110%、可変）
  - 通知の事前リマインド日数（デフォルト [3,1,0]、可変）
  - 並行運用フラグ（true で Sheets 連携 UI 表示）
- F-9.3 変更履歴は `audit_log` に記録

### FR-10 Drive 連携
- F-10.1 教師契約書 PDF は Google Drive 上に配置、`teachers.contract_drive_file_id` で参照
- F-10.2 アップロード UI から Drive へ直接アップロード
- F-10.3 表示時は短時間有効な署名付きリンクで配信
- F-10.4 **専用サービスアカウント**を別途発行（運用側で IAM・権限を通したもの）
  - 教師契約書フォルダのみ Editor 権限
  - 鍵は Cloud Secret Manager に格納

### FR-11 アクティブ／過去フィルタ
- F-11.1 一覧系画面はデフォルトで**アクティブのみ表示**
- F-11.2 画面上部に「過去分も表示」チェックボックス
- F-11.3 アクティブ判定:
  - 教師: `is_active=true AND status!='disabled'`
  - 生徒: `status IN ('enrolled','on_break')`
  - 指導枠: `status='active'`
- F-11.4 検索 API クエリ `?include_inactive=true` で切替（HTMX で再フェッチ）

## 4. 非機能要件（NFR）

| カテゴリ | 要件 |
|---|---|
| 性能（根拠付き） | 想定規模: 管理者10／教師100／生徒200／指導枠600。一覧画面 P95 < 800ms、月次集計 P95 < 2s |
| 同時接続 | ピーク 30 セッション程度 |
| データ保持 | **半永久**（卒業生徒も論理削除のみ） |
| 認証 | Supabase Auth Magic Link / 招待メール方式 |
| セキュリティ | RLS で `assignments.teacher_id = auth.user_teacher_id()` を強制 |
| ログ | 全 mutating 操作 + 設定変更を audit_log |
| バックアップ | Supabase 日次 + 週次論理ダンプを GCS（半永久対応） |
| 同時編集 | 楽観ロック（updated_at + If-Match） |
| ブラウザ | Chrome／Edge／Safari の最新2世代 |
| アクセシビリティ | WCAG 2.1 AA 努力目標 |
| i18n | 日本語のみ |

## 5. データモデル

```
teachers (id, teacher_no, name, email UNIQUE, university, ib_score,
          subjects_hl[], subjects_sl[], languages[],
          contract_status, contract_drive_file_id,
          bank_info_id, training_status, working_status,
          recommend_point,
          status TEXT CHECK (status IN ('invited','active','disabled')) DEFAULT 'invited',
          invited_at, activated_at,
          is_active BOOLEAN DEFAULT true, created_at)

students (id, student_no, name, grade, dp_phase, exam_term,
         enrolled_at, target_university, target_field,
         status, created_at)

bank_info (id, ...)  -- teachers から FK、経理ロールのみ可視

assignments (id, selection_no UNIQUE, student_id, teacher_id,
             course, subject, monthly_hours,
             started_at, ended_at, status,
             external_sheet_url)

assignment_history (id, assignment_id, teacher_id_from,
                    teacher_id_to, changed_at, reason)

monthly_reports (id, assignment_id, year_month,
                 actual_hours, fulfillment_rate,
                 report_submitted, student_achievement,
                 teacher_achievement, plan_quality,
                 cancel_count, satisfaction,
                 unit_test_score, submitted_at)
   -- UNIQUE (assignment_id, year_month)

meetings (id, student_id, kind, held_at, attendees[],
          body, source_hash, created_by, created_at)

meeting_tags (id, meeting_id, tag)

todos (id, meeting_id, student_id, assignee_id,
       title, due_date, status, completed_at)

users (id, supabase_uid, role, teacher_id?, name)

teacher_invitations (id, teacher_id, token, expires_at,
                     sent_at, sent_by, accepted_at, status)

import_jobs (id, source, status, started_at, finished_at,
             total_rows, success_rows, error_rows, run_by)

import_errors (id, job_id, sheet_name, row_no, raw_payload_json,
               error_message, resolved_at, resolved_by)

system_settings (key, value_json, updated_by, updated_at)
  -- 例: ('monthly_report_deadline', '{"type":"month_end"}'),
  --     ('fulfillment_rate_warn', '{"min":80,"max":110}'),
  --     ('reminder_days_before', '[3,1,0]'),
  --     ('parallel_operation_until', '{"date":"2026-08-31"}')

audit_log (id, user_id, action, entity, entity_id,
           diff_json, at)
```

**RLS ポリシー（議事録）**:
```sql
CREATE POLICY meetings_teacher_read ON meetings
FOR SELECT USING (
  auth_role() = 'admin'
  OR auth_role() = 'accounting'
  OR EXISTS (
    SELECT 1 FROM assignments a
    WHERE a.student_id = meetings.student_id
      AND a.teacher_id = auth_user_teacher_id()
      AND a.status = 'active'
  )
);
```

## 8. 確定済みの前提（PO 回答済）

- 並行運用期間: **3か月**、URL 紐付け必要
- 教師ログイン: **招待メール方式**（teachers.email 登録ユーザのみ）
- 月次レポート締切日: **設定画面で可変**
- 充足率異常値の閾値: **設定画面で可変**
- 議事録の機密度: **教師は自担当生徒のみ閲覧**
- 教師契約書 PDF: **Drive 連携**（別途 IAM/権限を通した SA を用意）
- データ保持: **半永久**
- 想定規模: 管理者10／教師100／生徒200
- Phase -1 サンプリング対象: **全タブ**
- 既存 Sheets 抽出方針: **自動パーサー（Phase -1 で仕様化済み）**
- 半永久保持の運用: **デフォルトはアクティブのみ表示、チェックボックスで過去分**
- 設定画面の権限分離: **保留（後日）**

## 9. 残課題（Phase 0 で詰める）

- C-1. 招待メール文面・差出人ドメインのドラフト
- C-2. 下部記録表 2 形式（原本 vs 三者面談・教師mtg）のうちどちらを正とするか
- C-3. 独自列で取込対象に含めるべきものがあるか（事業観点）
- C-4. 仕様書レビューの参加メンバー確定（原口・元山・藤井）
- C-5. 教師の苗字単独表記（`林` 等）が複数 canonical に該当する場合の扱い
- C-6. `Math(HL)` のレベル＆種別不明な科目表記の扱い
- C-7. 退職教師が canonical にない場合の運用
- C-8. 並行運用期間中の自動パーサー実行頻度（毎晩 / 週次）
- C-9. dryrun の差分閾値（書き換え件数の警告ライン）
- C-10. 設定画面の権限分離（締切日変更 vs 並行運用フラグ）
