## why（背景・目的）

**背景（As-Is）**

- 運用中の Google Sheets は **37〜38 タブ規模** に達し、保守・権限・一貫性の限界が見えている。
- 教師・生徒・契約・月次評価・面談記録が同一ブックに集中しており、ロールベースのアクセス制御と監査のしやすさが課題。

**目的（To-Be）**

- **FastAPI + Jinja2 + HTMX + Supabase + Cloud Run** による業務用 Web アプリへ移行し、**Supabase RLS** で「教師は自担当のみ」等を DB 層で強制する。
- **3 か月の並行運用**ののち Sheets を凍結・廃止し、**半永久的なデータ保持**（論理削除・アクティブフィルタ）を維持する。

**成功状態（卒業要件のイメージ）**

- 主要業務（教師・生徒・指導枠・月次レポート、および当面必要な面談・通知）が Web で完結し、Sheets 依存がない。
- PO による受け入れレビュー完了。
- 移行に関するオープンポイント（教師名ゆれ、科目表記、インポート頻度等）が文書化された方針に沿って実装されている。

**歓迎要件**

- 既存 Sheets 行へのワンクリック遷移（並行運用期間）。
- 設定画面での締切日・充足率閾値・リマインド日数の動的変更。

## what・How（内容・方法）

- **機能スコープ**: [docs/requirements/v0.3.md](../requirements/v0.3.md) の IN に従う。OUT（保護者ポータル、請求など）はフェーズ 2 以降。
- **認証**: 教師は **招待メール方式**（セルフサインアップ不可）。管理者が `teachers` 登録後に `inviteUserByEmail()`。トークン **72h**。
- **Sheets 移行**: Phase -1 成果の [parser-spec.md](../sheets-migration/parser-spec.md) に基づき Phase 3 で自動パーサー実装。解釈不能行は `import_errors` で管理。
- **Drive**: 教師契約書 PDF は **専用 SA**（IAM 整理済み）で連携。既存調査用 SA とは切り分け。
- **UI 方針**: モック先行（`spec-driven-mock-ui` に準拠）、ダッシュボードは「自分宛て」優先。
