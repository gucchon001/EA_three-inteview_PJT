# T1-3 セキュリティ要件定義書
## HTMLレポート配布における個人情報保護要件

**タスクID**: T1-3  
**作成日**: 2026年4月2日  
**ステータス**: 決定済み  
**関連ドキュメント**: [技術的タスクリスト_アクションアイテム対応.md](技術的タスクリスト_アクションアイテム対応.md)

---

## 1. 採用方式の決定

### ワンタイムURL方式（署名付きURL）

保護者へのレポート配布は、**サーバーサイドで生成・管理するワンタイムURLをメール送信する方式**を採用する。

```
【配布フロー】
CAがシステムからトークン発行
    ↓
署名付きURL（ワンタイムトークン付き）をメール送信
    ↓
保護者がURLをクリックして閲覧
    ↓
有効期限切れ後は自動失効
```

---

## 2. 技術要件

### 2-1. トークン仕様

| 項目 | 仕様 |
|---|---|
| トークン長 | 128bit以上のランダム文字列（UUID v4 相当） |
| 有効期限 | 発行から **14日間** |
| 使用回数制限 | 制限なし（期限内は何度でも閲覧可） |
| 失効条件 | 有効期限切れ、または手動失効（再発行時に旧トークンを自動失効） |

### 2-2. URL構造

```
https://[ドメイン]/report?token=[128bit乱数トークン]
```

- トークンからレポートIDや生徒情報は**推測不可能**であること
- URLにレポートID・生徒番号・氏名等の個人情報を含めないこと

### 2-3. サーバーサイド要件

| 要件 | 詳細 |
|---|---|
| HTTPS必須 | 全通信をTLS暗号化。HTTPへのアクセスはHTTPSにリダイレクト |
| コンテンツの分離 | HTMLにレポート本文を含めない。トークン検証後にAPIでコンテンツを返却 |
| トークン検証 | 毎リクエストでサーバー側でトークンの有効性・期限を検証 |
| アクセスログ | IPアドレス・日時・トークンIDをログに記録（最低6ヶ月保存） |

---

## 3. 再発行フロー

有効期限切れ後に保護者が再閲覧を希望する場合、以下のフローで対応する。

```
【再発行フロー】
保護者 → 担当CA（登録済み連絡手段）に再発行を依頼
    ↓
CA が本人確認を実施（下記ルール参照）
    ↓
CA がシステムから再発行操作（旧トークンは自動失効）
    ↓
新しいワンタイムURLをメールで送信
```

### 再発行時の本人確認ルール

| 依頼手段 | 確認方法 |
|---|---|
| LINE（登録済みアカウント） | 登録済みLINEからの依頼であることを確認 |
| 電話 | 登録済み電話番号からの着信であることを確認 |
| メール | 登録済みメールアドレスと送信元が一致することを確認 |
| 上記以外 | 対応不可。登録済み連絡手段からの依頼を案内する |

> **重要**: 未登録の連絡手段からの再発行依頼は、本人確認ができないため対応しない。

---

## 4. メール送信の運用ルール

URLの技術的安全性に加え、**誤送信が最大のリスク**であるため、以下の運用ルールを必須とする。

| ルール | 詳細 |
|---|---|
| 送信前の宛先確認 | システムに登録されているメールアドレスを画面で確認してから送信 |
| 二重確認ダイアログ | 送信前に「氏名・メールアドレス・レポート対象月」を表示して最終確認 |
| 誤送信時の即時対応 | 誤送信を検知した場合、CAがシステムから該当トークンを即時失効させる |
| メールアドレス変更 | 保護者からの変更依頼は本人確認後のみ受け付ける |

---

## 5. アクセスログの仕様

| ログ項目 | 内容 |
|---|---|
| 日時 | アクセス日時（JST） |
| トークンID | アクセスに使用したトークンの識別子 |
| IPアドレス | アクセス元IP |
| 結果 | 成功 / 期限切れ / 無効トークン |

- ログは6ヶ月以上保存
- 不正アクセス（無効トークンへの連続アクセス等）はアラート通知

---

## 6. 個人情報保護法との整合性

| 安全管理措置 | 対応状況 |
|---|---|
| 技術的安全管理（アクセス制御） | ○ トークン認証・HTTPS・コンテンツ分離 |
| 技術的安全管理（アクセスログ） | ○ ログ記録・保存 |
| 組織的安全管理（運用ルール） | ○ 誤送信防止・本人確認ルール |
| 漏洩時の対応手順 | ○ 誤送信時のトークン即時失効フロー |

**結論**: 上記要件をすべて満たした実装・運用を行うことで、個人情報保護法の安全管理措置として適切と判断する。

---

## 7. 未対応リスクと許容判断

| リスク | 対策 | 許容判断 |
|---|---|---|
| URLの転送・共有 | 有効期限14日で被害範囲を限定 | 許容（塾業界の月次レポートとして標準的） |
| メールサーバーへの保存 | TLS対応サービス（Gmail等）を使用 | 許容 |
| ブラウザ履歴へのURL残存 | 閲覧後もトークンは有効（再閲覧を優先） | 許容（要期限切れによる自動失効） |
