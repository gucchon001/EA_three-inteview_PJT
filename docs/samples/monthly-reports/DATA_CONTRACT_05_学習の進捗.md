## 1. 用語と不変条件（契約のコア）

| 用語 | 定義 |
|------|------|
| **レーン** | 1 トピックあたり **学校** と **当塾** の 2 行（比較単位）。 |
| **マス** | 1 レーンを **10 セル**に等分したもの。左から右へ時間・進捗の進行方向とする（実装は CSS グリッド等でよい）。 |
| **状態** | 各マスは次のいずれか **1 つだけ**: `done`（済・青） / `planned`（直近予定・黄） / `pending`（未・灰）。 |
| **正本（canonical）** | レポート生成・レビューで参照するのは **`planned` を含む最終的な 10 マスの配列**（後述スキーマ）。% は派生表示用に任意。 |

**色（家庭向けラベルと実装クラス名の対応）**

| ラベル | `state` | 備考 |
|--------|---------|------|
| 済 | `done` | 当該トピック範囲として「完了」と判断できる単位が埋まっているイメージ。 |
| 直近1ヵ月の予定 | `planned` | **未済**だが、下記ウィンドウ内に授業・復習・テスト等の**予定が計画表上で示されている**部分。家庭向け文言は **「レポート対象月のあいだの予定」** と注記すると、既定ウィンドウ（案 B）と一致してわかりやすい。 |
| 未 | `pending` | 上記以外の未着手・未計画。 |

## 2. 「黄（planned）」の日付ウィンドウ（一次ルール）

レポートの **安定性**（毎回同じ境界で機械・人間が判断できること）と **見やすさ**（黄が増えすぎず青・灰との対比がはっきりすること）を優先し、次を **推奨既定** とする。

**推奨既定（案 B）**: レポートに明記する **対象月 `M`**（例: 「2026年4月の学習の様子」→ `M = 2026-04`）を基準にする。

- **ウィンドウ `W`**: **`M` の暦月 1 日 00:00 から、その暦月の末日 23:59:59 まで**（ inclusive ）。**翌月にまたがる予定は黄に含めない**（必要なら 06 の予定表で示す）。
- **黄にできるのは**、上記期間内に日付または明示文言がある **未済** のみ。すでに `done` とみなす範囲には黄を置かない。
- **根拠データ**: 学習計画表の **授業日・予定列・メモ**（および運用で許可するなら 06 の予定表）。日付が空なら黄は付けない。
- **家庭向け注記（推奨）**: 05 の凡例または脚注に **「黄色は、本レポートの対象月のあいだに予定がある範囲です」** を 1 行入れる（「直近1ヵ月」と書きたい場合も、実体は `M` の暦月と一致させる）。

**案 B を推奨する理由（要約）**

| 観点 | 案 B |
|------|------|
| 安定 | 境界が **暦月の固定端**のみで、発行日や PDF 出力タイミングに依存しない。 |
| 見やすさ | 黄が **最大でも「未済帯」の一部**に収まりやすく、帯が長く伸びて灰が読みにくくなりにくい。 |
| 文言との一致 | レポートタイトルの「◯月」と **同じ月**を指せる。 |

**代替案 A（広いウィンドウ）**: **`M` 月初から `M` の翌暦月の末日まで**を黄の対象とする。**翌月分まで同じバーに色を乗せたい**ときに限定採用。黄が増え、対象月とズレた印象になりやすいので **既定にはしない**。

**代替案 C（発行日基準）**: PDF 出力日から rolling 30 日。**月次レポートの「対象月」とズレやすい**ため、原則採用しない（特別用途のみ）。

> **既定**: 実装・レビュー・JSON 省略時は **`planned_window_rule`: `B`** とする。案 A が必要な生徒月のみ JSON で `A` を明示する。

## 3. 一次情報が「済/未だけ」のときの割合ルール（優先順位）

学習計画表が **トピック単位の TRUE/FALSE や 済/未 のみ**の場合、**10 マスに黄を混ぜた表現は自動では一意に決まらない**。次の **優先順位**で「データ供給モード」を選ぶ。

| 優先 | モード名 | 条件 | 10 マスの決め方（要約） |
|------|----------|------|-------------------------|
| 1 | **`explicit_lane`**（推奨・手動または拡張列） | スプレまたは `sources` に **レーンごとに 10 文字または JSON** がある | その値を正とする。機械は変換しない。 |
| 2 | **`subtopic_ratio`** | 同一トピック配下に **サブ行（小単元）**があり、各行に済フラグがある | `done` マス数 = round(10 × 済行数/总行数)。残りを **セクション4** の左から順のルールで `planned` / `pending` に配分。 |
| 3 | **`boolean_topic`**（フォールバック） | トピックに **1 ビットのみ** | `TRUE`/`済` → 10 マスすべて `done`。`FALSE`/`未` → **すべて `pending`**（黄は **セクション2** のウィンドウ内に予定日が明示されているときのみ、担当が `explicit_lane` で上書き）。 |

### 3.1 補足モード **`calendar_buckets_10`**（日次ベース・当塾授業日）

トピックの済/未だけでは **「授業のない日」** が見えない。学習計画表または **03 の「授業日」列**から **日付**が取れるときは、次のモードで **同じ 10 マス**を「月のカレンダーを圧縮したレーン」として使える（HTML サンプル **v3.2** では **トピック別**に採用）。

| 項目 | ルール |
|------|--------|
| 区間分割 | 対象月 `M` の日数を **10 区間**に分ける（例: 30 日 → 各 3 日、31 日 → 最終区間だけ 4 日など、端数は**末日側の区間**に吸収）。 |
| **当塾** のマス `i` | 区間 `i` に **03（または計画表）の当塾授業日が 1 日でも含まれる** → `done`。それ以外 → `pending`。 |
| **黄** `planned` | 区間内に **計画上の予定日**があり、かつ **まだ 03 に実績行が無い**（未実施の予定）で、かつ日付が **セクション2 案 B のウィンドウ**内。 |
| **学校** の同バー | 計画表に**学校の日次**が無い場合は **`pending` のまま**（誤解防止のため脚注で「週次・トピックは上表」と明示）。日次が取れるなら当塾と同様に埋める。 |

**トピック別に並べる場合（v3.2 HTML サンプル）**

- 05 の**各行（トピックまたは週ブロック）**ごとに、`prog-topic` を分け、**その行に対応する 03 授業日だけ**を集めて当塾レーンの区間を青にする。  
- 03 の「学習単元」列を 05 のトピック行へ割り当てるルールは **担当定義**（例: ある適合済み事例では Number/Algebra ← Geometric〜Factorization、Functions ← Quadratic functions、Geometry/Trig ← Trigonometry）。`sources` に 1 行メモを残すと再現性が上がる。

> **読み取りミスとの切り分け**: 旧 v3 の「トピックごとに 10 マス全部青」は、**データがすべて済**だったことに加え、**`boolean_topic` 的な割り当て**をしていたため。`calendar_buckets_10` に切り替えると、**同じ 03 日付**から灰マスが出る。

**丸め**: `subtopic_ratio` で端数が出る場合は **「済」マス数を四捨五入**し、10 を超えないようクリップする。残マスは **先に `planned` を黄ルールで埋め、残りを `pending`**（セクション4）。

## 4. 同一レーン内での状態の並べ方（左→右）

1. 左から **`done` を連続配置**（済の「塊」は左寄せ）。  
2. 続く領域に **`planned` を連続配置**（予定が可視化される「これからの帯」）。  
3. 右端側に **`pending`**。

**% 表記との関係**: 外部から `done_pct` / `planned_pct` が来る場合は、まず **マス数**に換算してから上記順で並べる。

- `n_done = round(10 * done_pct / 100)` など（合計が 10 を超えないよう `planned` → `pending` の順で 1 マスずつ削減）。

## 5. 正本 JSON スキーマ（案）

レポート 1 名分の 05 ビジュアル用。フィールド名は実装で snake_case 固定（HTML 生成スクリプトと共有）。

```json
{
  "section": "05_learning_progress",
  "report_anchor_month": "2026-04",
  "planned_window_rule": "B",
  "topics": [
    {
      "id": "topic_1",
      "label": "Topic A（表示名）",
      "school": { "cells": ["done","done","planned","pending","pending","pending","pending","pending","pending","pending"], "source_mode": "explicit_lane" },
      "cram":   { "cells": ["done","done","done","done","done","done","done","pending","pending","pending"], "source_mode": "subtopic_ratio" }
    }
  ]
}
```

**制約**

- `cells` は **長さちょうど 10**。要素は `"done"` / `"planned"` / `"pending"` のみ。  
- `source_mode` は **`explicit_lane` | `subtopic_ratio` | `boolean_topic`** のいずれか（トレーサビリティ用）。  
- `planned_window_rule` は **`B` | `A` | `C`**（セクション2。**既定は `B`**）。未記載は **`B`** とみなす。

## 6. スプレッドシート拡張（任意・運用が楽になる列）

学習計画表に列を足せる場合の例（トピック行またはサブトピック行に応じて 1 行ずつ）:

| 列名（例） | 内容 |
|------------|------|
| `progress_05_school_cells` | 10 文字。`D`=done, `P`=planned, `N`=pending（ASCII で入力しやすくするため）。 |
| `progress_05_cram_cells` | 同上。 |

インポート時に `D`→`done` 等へマップする。**正本は常に 10 マス配列**とする。
