# 要件定義書

## 位置づけ

- 正本/補助資料の区分: 月次レポート作成ツールの要件定義
- 起点: `docs/project/月次レポート_プログラム化_LLMワークフロー移行計画.md`
- 関連文書: `README.md`, `functional-spec.md`, `security-operations.md`, `development-plan.md`
- 最終更新: 2026-05-13

## 目的

月次レポート作成をエージェントやチャット内の暗黙知から切り離し、ブラウザ上でデータソースの指定・取得・確認・LLM生成・規約検証・推敲まで進められる業務ツールに移行する。

MVPでは配布完成度よりも、実運用を回しながらプロンプト、テンプレート、コード、LLMモデルの組み合わせをチューニングできる状態を優先する。

## スコープ

### MVPに含む

- Googleアカウントログイン
- `tomonokai-corp.com` ドメイン限定
- Google Sheets / Docs / Drive APIからのユーザーOAuthによるサーバ取得
- 対象月・対象世帯・ソース指定
- ソース取得結果の確認
- 非同期ジョブによるレポート草稿生成
- OpenRouter経由のLLM呼び出し
- Markdown草稿の保存
- 決定的バリデーション
- 検証結果の表示
- ソーススナップショット、生成物、検証結果、LLMメタ、版情報の保存
- 生成結果への人手フィードバック保存
- 1ユーザー最大3件の同時生成ジョブ制限
- Cloud Run本番環境での稼働

### MVPでは任意または後続

- HTMLエクスポート
- 既存静的配信経路への自動同期
- プロンプト版・モデル版のUI切替
- repair loopの自動実行
- SendGrid等によるメール配信
- 保護者向け閲覧ポータル
- Supabaseポータルへの完全統合

### スコープ外

- ブラウザへのサービスアカウント鍵配置
- フロントエンドからOpenRouter APIを直接呼ぶ実装
- CSR専用SPAへの置き換え
- Vertex / Gemini直利用への即時切替

## 機能要件

| ID | 要件 | MVP |
|---|---|---|
| FR-1 | ユーザーはSupabase AuthのGoogle providerでログインできる | 必須 |
| FR-2 | ログイン可能なメールドメインは `tomonokai-corp.com` に限定する | 必須 |
| FR-3 | ユーザーはGoogle Sheets / DocsのURLまたはIDを指定できる | 必須 |
| FR-4 | サーバはユーザーOAuthに基づきSheets / Docs / Drive APIでソースを取得する | 必須 |
| FR-5 | 取得したソースの一覧、種別、サイズ、取得状態を確認できる | 必須 |
| FR-6 | ユーザーは対象月・対象世帯・テンプレート種別を指定して生成ジョブを作成できる | 必須 |
| FR-7 | 生成ジョブは非同期で実行し、状態をポーリング表示できる | 必須 |
| FR-8 | ジョブは fetch, bundle, build_messages, call_llm, validate, persist の段階を持つ | 必須 |
| FR-9 | 生成結果はMarkdownを第一成果物として保存する | 必須 |
| FR-10 | 必須見出し、禁止語、星5行原文一致などを決定的に検証する | 必須 |
| FR-11 | 検証エラー・警告をUIに表示する | 必須 |
| FR-12 | ユーザーは生成結果に問題カテゴリと自由記述フィードバックを残せる | 必須 |
| FR-13 | 推敲画面は現行HTML全文エディタをベースに、iframe編集・HTMLソース直接編集・ファイル保存・送付エクスポートを提供する | MVP内必須 |
| FR-14 | 同一ソーススナップショットから再生成できる | 後続優先 |
| FR-15 | 管理者はチューニング用途でモデルやテンプレート版を指定できる | 後続優先 |

## 非機能要件

| ID | 要件 | 内容 |
|---|---|---|
| NFR-1 | 再現性 | ジョブごとにソーススナップショット、prompt_version、template hash、モデルID、Git SHA相当を保存する |
| NFR-2 | 監査性 | 生成・再生成・フィードバック・設定変更を監査ログに残す |
| NFR-3 | セキュリティ | APIキー、OAuthトークン、ソース本文、生成物を権限管理する |
| NFR-4 | PII保護 | Cloud Loggingには本文・氏名・スコアなどを生出力しない |
| NFR-5 | 同時実行 | 1ユーザー最大3生成ジョブ |
| NFR-6 | 可用性 | MVPは本番のみ。復旧手順と失敗時の再実行導線を持つ |
| NFR-7 | 保守性 | プロンプト断片はリポジトリ内ファイルで管理し、コード内長文字列を避ける |
| NFR-8 | 拡張性 | 将来のSupabaseポータル統合を見越してルータ・サービス・テンプレートを分離する |
| NFR-9 | データ保持 | ソーススナップショットは1年、ジョブメタ・生成/最終編集Markdown・検証結果・フィードバックは3年、構造化ログは6か月保持する |
| NFR-10 | 永続化基盤 | MVPからSupabase Postgresを採用する |
| NFR-11 | Cloud Run初期構成 | リージョンは `asia-northeast1`（東京）。timeout 900秒、1 vCPU、1GiB、concurrency 10、min instances 0、max instances 3 |
| NFR-12 | OAuthスコープ | `openid`, `email`, `profile`, Sheets readonly, Docs readonly, Drive readonly。MVPでは書き込みスコープを持たない |
| NFR-13 | ローカル認証 | ローカル環境ではGoogle/Supabase Authの実ログインを使わず、明示的に有効化したモック認証を使う |

## 成功基準

- 実案件の入力から、ブラウザ操作でMarkdown草稿を生成できる。
- 同一ジョブの入力・モデル・プロンプト・テンプレート版を後から確認できる。
- 検証結果と人手フィードバックを、チューニング材料として蓄積できる。
- 生成処理が失敗しても、どの段階で失敗したか追跡できる。
- MVP中に少なくとも3件以上の実案件で、生成結果と修正内容を比較できる。
